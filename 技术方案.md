#### 3.1 预处理模块（app/modules/preprocessor.py）

3.1.1 职责
- 解析原始消息 JSON，提取纯文本。
- 解析 `members` 字段构建群成员映射。
- 补全发送者角色（销售/售后/客户）。
- 输出标准化消息对象。

3.1.2 角色推断逻辑

根据您提供的信息，`job` 字段可能为空，且 `position` 字段未完全覆盖角色。设计如下推断规则（按优先级）：

1. 若 `job` 字段值为 `"销售"`、`"售后"`、`"客户"` → 直接使用。
2. 若 `job` 为空，检查 `position` 字段：
   - 包含“销售” → 角色 `"销售"`
   - 包含“技术”、“售后”、“支持” → 角色 `"售后"`
   - 其他 → 暂标记为 `"未知"`，后续根据 `members.type` 判断。
3. 若以上均无法确定，检查 `members` 中 `type` 字段：
   - `type=2` → 角色 `"客户"`
   - `type=1` → 角色 `"员工"`（无法细分销售/售后时使用）
4. 若仍无法确定 → 角色 `"未知"`

3.1.3 成员缓存实现

使用 `cachetools.TTLCache` 实现带过期时间和最大容量的缓存：

from cachetools import TTLCache

    def set_room_members(self, room_id: str, members: dict):

3.2.3 规则匹配实现

使用 `pyahocorasick` 构建 AC 自动机，在应用启动时初始化，全请求复用：

```python
class SentimentRuleMatcher:
    def init(self):
        self.automaton = ahocorasick.Automaton()
        self.word_type_map = {}  # word -> 'severe_negative' / 'negative' / 'positive'

    def build(self, word_dict: dict):
        for word_type, words in word_dict.items():
            for word in words:
                self.automaton.add_word(word, (word, word_type))
        self.automaton.make_automaton()

    def match(self, text: str) -> tuple:
        """返回 (sentiment, confidence, matched_words)"""
        hits = []
        for end_idx, (word, word_type) in self.automaton.iter(text):
            hits.append((word, word_type))
        # 优先返回严重负面
        for word, wtype in hits:
            if wtype == "severe_negative":
                return ("very_negative", 0.95, [word])
        if hits:
            # 根据第一个命中决定倾向
            wtype = hits[0][1]
            sentiment = "negative" if wtype == "negative" else "positive"
            return (sentiment, 0.80, [w for w, _ in hits])
        return (None, 0.0, [])
```

3.2.4 LLM 调用设计（通义千问）

- 模型：`qwen-turbo`（性价比高）或 `qwen-plus`（准确率更高）
- Prompt 模板：

```text
你是一个企业微信客户沟通质量监控助手。请判断以下消息的情感倾向，输出严格的 JSON 格式，不要包含任何额外文本。

消息内容：{text}

输出格式：
{"sentiment": "positive/neutral/negative/very_negative", "confidence": 0.0-1.0, "is_abusive": true/false, "reason": "简短理由"}
```

- 并发控制：使用 `asyncio.Semaphore(LLM_MAX_CONCURRENT)` 限制同时进行的 LLM 请求数，防止 API 限流。
- 重试机制：失败时指数退避重试最多 2 次，超时时间 30 秒。
- 输出解析：使用正则提取 JSON 代码块，并做字段默认值填充：

```python
import re
import json

def parse_llm_response(text: str) -> dict:
    # 提取 JSON
    match = re.search(r'\{[^{}]*\}', text, re.DOTALL)
    if not match:
        return {"sentiment": "neutral", "confidence": 0.5, "is_abusive": False}
    try:
        data = json.loads(match.group())
    except:
        return {"sentiment": "neutral", "confidence": 0.5, "is_abusive": False}

    return {
        "sentiment": data.get("sentiment", "neutral"),
        "confidence": float(data.get("confidence", 0.5)),
        "is_abusive": bool(data.get("is_abusive", False)),
        "reason": data.get("reason", "")
    }
```

#### 3.3 敏感词检测模块（app/modules/sensitive.py）

3.3.1 敏感词库加载

您提供的敏感词列表将存储于 `app/resources/sensitive_words.txt`，每行一个词。应用启动时加载并构建 AC 自动机。

3.3.2 归一化预处理

匹配前对文本进行以下归一化，防止简单绕过：
- 移除所有空格、制表符、换行符
- 全角符号转半角（如 `，` → `,`）
- 繁体转简体（可选，使用 `zhconv` 库）

3.3.3 匹配逻辑

```python
class SensitiveDetector:
    def init(self, word_file: str):
        self.automaton = ahocorasick.Automaton()
        with open(word_file, 'r', encoding='utf-8') as f:
            for line in f:
                word = line.strip()
                if word:
                    self.automaton.add_word(word, word)
        self.automaton.make_automaton()

    def detect(self, text: str) -> list:
        if not text:
            return []
        # 归一化
        normalized = re.sub(r'\s+', '', text)
        hits = []
        for end_idx, word in self.automaton.iter(normalized):
            if word not in hits:  # 去重
                hits.append(word)
        return hits
```

3.4.1 消息预处理流程

1. 清洗：过滤满足以下条件的消息：
   - `content` 长度 < 2
   - 纯表情（正则匹配）
   - 系统消息（`msgtype` 为 `Event` 或 `sys`）
2. 采样（若消息数 > 150）：
   - 强制保留：所有包含 `?`、`？` 的消息（疑问句，信息量大）。
   - 强制保留：情感标签为 `negative` 或 `very_negative` 的消息。
   - 均匀采样：剩余消息按时间等间隔选取，使总数 ≤ 150。
3. 格式化：
   ```
   [{role}]{name}: {content}
   ```
   例如：`[售后]高鹏博: 其实如果咱们宿主占比不高的话 影响不是很大的`

3.4.2 Token 计数与截断

使用通义千问官方 Tokenizer 或保守估算（中文约 1.5 字符/Token）。

- 上下文限制：`qwen-turbo` 支持 8K Token，`qwen-plus` 支持 32K。
- 预留 500 Token 给 Prompt 模板和响应。
- 若拼接后消息超过限制，从最早的消息开始截断（保留最近的消息）。

```python
def truncate_by_tokens(messages: list, max_tokens: int = 7000) -> list:
    total = 0
    result = []
    for msg in reversed(messages):  # 从最新往前加
        token_estimate = len(msg["content"]) // 1.5 + 20  # 20 为角色前缀 Token
        if total + token_estimate > max_tokens:
            break
        result.insert(0, msg)
        total += token_estimate
    return result
```

3.5.3 分词与统计

- 每条消息分词后过滤停用词、单字词、纯数字。
- 文档频率（DF）：同一词语在同一消息中出现多次，只计一次次数（避免刷榜）。
- 来源记录：使用 `collections.deque(maxlen=5)` 存储最近 5 条来源，控制内存。

```python
from collections import defaultdict, deque

class HighFreqAnalyzer:
    def init(self, dict_path, stopwords_path):
        jieba.load_userdict(dict_path)
        with open(stopwords_path, 'r') as f:
            self.stopwords = set(line.strip() for line in f)

    def process(self, messages: list) -> list:
        word_stats = defaultdict(lambda: {"count": 0, "sources": deque(maxlen=5)})

        for msg in messages:
            if not msg.content:
                continue
            words = jieba.lcut(msg.content)
            valid_words = set()
            for w in words:
                if len(w) > 1 and w not in self.stopwords and not w.isdigit():
                    valid_words.add(w)

            source = {
                "sender_name": msg.sender_name,
                "sender_position": msg.sender_position,
                "sender_job": msg.sender_job,
                "room_id": msg.roomid,
                "msgtime": msg.msgtime.isoformat()
            }

            for w in valid_words:
                word_stats[w]["count"] += 1
                word_stats[w]["sources"].append(source)

        # 排序取 Top10
        sorted_words = sorted(word_stats.items(), key=lambda x: x[1]["count"], reverse=True)[:10]
        return [
            {
                "word": word,
                "frequency": data["count"],
                "sources": list(data["sources"])
            }
            for word, data in sorted_words
        ]
```

### 5. LLM 调用策略详细设计

#### 5.1 通义千问客户端封装（app/core/llm_client.py）

- 单例模式：全局复用 `httpx.AsyncClient` 连接池。
- 并发控制：`asyncio.Semaphore` 限制最大并发请求。
- 重试策略：指数退避，首次重试等待 1 秒，第二次 2 秒。
- 超时设置：连接超时 10 秒，读取超时 30 秒。

#### 5.2 成本估算

以日均分析 1000 个群、每群 100 条消息（采样后）为例：

| 场景 | 调用次数 | Token 消耗估算 | 月成本（qwen-turbo） |
| :--- | :--- | :--- | :--- |
| 情感分析（需 LLM 的消息约占 30%） | 1000 × 100 × 0.3 = 30,000 次 | 每次约 200 Token（含 Prompt+响应） | 约 ¥0.8/日，¥24/月 |
| 摘要生成 | 1000 次 | 每次约 1500 Token | 约 ¥0.2/日，¥6/月 |
| 合计 | | | 约 ¥30/月 |

注：通义千问 qwen-turbo 定价为输入 0.008 元/千 Token，输出 0.024 元/千 Token。成本极低。
