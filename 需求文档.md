1. 项目背景与目标

1.1 项目背景
公司销售与售后团队通过企业微信与客户建立大量服务沟通群。群内沟通质量直接影响客户满意度与公司声誉。目前缺少对群聊内容的自动化监控与分析手段，难以主动发现沟通风险（如态度恶劣、违规承诺）、把握客户关注热点及服务动态。

1.2 项目目标
建设一套基于大语言模型（LLM）的企业微信群聊智能分析服务，实现对群聊内容的自动化、结构化分析。具体目标如下：
1. 风险预警：自动识别员工与客户沟通中的负面情感及恶劣对话，防止事态升级。
2. 合规监控：统计预设敏感词的触发情况，确保沟通内容符合公司规范。
3. 动态洞察：自动生成群聊信息摘要，帮助管理者快速了解客户群体动向。
4. 热点挖掘：统计高频词并追溯发言者身份，洞察客户真实关注点。
  
2. 系统范围与角色

2.1 系统范围
本系统作为 Python 开发的独立分析微服务，接收由 Java 后端提供的企业微信群聊消息数据，经过智能分析后，将结构化结果返回给 Java 后端，由 Java 后端负责数据存储、展示及告警推送。

2.2 用户角色
- Java 后端系统：主要调用方，负责数据采集、任务调度、结果展示。
- 业务管理者/质检人员：间接用户，通过前端界面查看分析报表与预警信息。
  
3. 功能性需求

3.1 数据接收与预处理 (FR-01)
- FR-01-01：系统应提供 RESTful API，接收 Java 侧推送的单群或多群消息列表。
- FR-01-02：系统需解析原始消息中的嵌套 JSON 内容，提取纯文本。
- FR-01-03：系统需解析群成员信息（members 字段），构建发送者身份映射表，补全发送者姓名、职位、角色（客户/售后/销售等）。
  
3.2 分角色情感分析 (FR-02)
- FR-02-01：系统需对每条文本消息进行情感倾向分析，分类为：positive（积极）、neutral（中性）、negative（负面）、very_negative（恶劣）。
- FR-02-02：分析结果需关联发送者角色（客户、售后、销售），以便统计各角色情绪分布。
- FR-02-03：当检测到 very_negative 或恶劣对话时，需在返回结果中标注该消息详情，供上游系统触发预警。
- FR-02-04：支持基于规则的关键词快速判断，以降低大模型调用成本；对于规则无法覆盖的复杂语义，调用 LLM 进行深度分析。
  
3.3 敏感词检测 (FR-03)
- FR-03-01：系统需维护一组可配置的敏感词列表。
- FR-03-02：检测消息中是否包含敏感词，返回命中的词列表及对应的消息发送者信息。
- FR-03-03：支持对敏感词库的动态更新（通过配置中心或环境变量）。
  
3.4 群聊信息摘要生成 (FR-04)
- FR-04-01：根据指定时间段内的群聊消息，调用 LLM 生成一段 200-300 字的中文摘要。
- FR-04-02：摘要内容应覆盖：主要讨论话题、客户情绪趋势、需关注的问题点。
- FR-04-03：生成摘要前需对原始消息进行清洗与采样，移除无效信息，确保输入不超过 LLM 上下文限制。
  
3.5 高频词统计与溯源 (FR-05)
- FR-05-01：统计群聊中出现频率最高的 Top10 关键词。
- FR-05-02：每个高频词需附带来源信息，包括：发言人姓名、发言人职位、发言人角色（客户/员工）、所属群组 ID、发言时间。
- FR-05-03：分词过程需过滤停用词（如“的”、“了”）和无意义词，支持加载业务自定义词典。
  
3.6 接口响应 (FR-06)
- FR-06-01：分析接口返回统一的 JSON 结构，包含请求的群组信息、统计时间范围、情感分析汇总、敏感词命中详情、摘要文本、高频词列表。
- FR-06-02：接口应具备幂等性，相同请求多次调用返回一致结果（或明确标记为重复处理）。
  
4. 非功能性需求

4.1 性能要求
- NFR-01：单次请求（含 100 条消息）的端到端分析延迟应控制在 5 秒以内（不含 LLM 服务自身延迟波动）。
- NFR-02：支持高峰期每分钟处理 10 个以上分析请求。
  
4.2 可扩展性
- NFR-03：模块化设计，各分析功能（情感、敏感词等）支持独立启用或禁用。
- NFR-04：LLM 服务应支持灵活替换（如从 OpenAI 切换至国内模型 DeepSeek 或私有化模型），通过配置实现。
  
4.3 可靠性
- NFR-05：当 LLM 服务调用失败时，系统应降级返回基于规则的分析结果或明确错误码，不应导致整个分析请求失败。
- NFR-06：系统应记录详细的请求与响应日志，便于问题追踪。
  
4.4 安全性
- NFR-07：敏感词库内容不得硬编码在源代码中，需通过安全的环境变量或配置中心注入。
- NFR-08：接口应部署在内网环境，仅允许 Java 后端服务访问。
  
5. 数据示例说明

5.1 输入数据样例（由 Java 提供）
参考需求中给出的 JSON 格式，主要关注字段：
- from：发送者 ID
- truename / position / job：身份信息
- content：内含实际文本的嵌套 JSON
- members：群成员详情列表
- msgtime：消息时间
  
5.2 输出数据样例（返回给 Java）
参考响应体结构，包含 sentiment、sensitive_words、summary、high_freq_words 等结构化数据块。

6. 验收标准
1. 接口联调成功，Java 侧能正确接收并解析 Python 返回的分析报告。
2. 情感分析准确率在人工抽检的 100 条样本中，严重负面（恶劣）识别准确率 > 90%。
3. 敏感词命中功能实测准确无误报（允许少量因分词导致的误报，需记录）。
4. 摘要内容通顺、符合事实，无明显幻觉内容。
5. 高频词统计结果可追溯到具体发言者，来源信息准确。
