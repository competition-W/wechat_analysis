## 更新日志

### V1.2 (2026-07-17)

**新增功能：**
- 新增看板通用下钻预览接口 `GET /api/v1/dashboard/drilldown`
- 新增当前数据点明细 Excel 导出接口 `GET /api/v1/dashboard/drilldown/export`
- 下钻接口统一沿用看板日期、区域、售后、产品和重点客户筛选口径
- 消息型明细默认脱敏手机号和企业微信用户标识

### V1.1 (2026-04-22)

**新增功能：**
- 新增 `unanswered` 分析类型，支持漏回检测功能
- 新增多个请求字段：`resultAsrTime`, `filepath`, `openId`, `reAvatar`, `rePosition`, `isSingle`, `rawJson`

**结构变更：**
- **情感分析结果重构**：从四分类(positive/neutral/negative/very_negative)改为按角色分类
  - 客户：`good_reviews`(好评)、`bad_reviews`(差评)
  - 员工：`positive`(积极)、`bad_attitude`(恶劣态度)
  - 移除 `alerts` 字段，新增 `details` 字段包含详细消息列表
  
- **高频词分析升级**：从简单词频统计改为业务需求提取
  - 返回标准化业务名称
  - 包含客户原始表述别名列表
  - 移除 `sources` 字段

**响应字段新增：**
- 新增 `unanswered_status` 字段，包含漏回检测结果

## 3. 群聊分析接口（单个）

### 3.1 接口信息

| 项目 | 值 |
| :--- | :--- |
| URL | /api/v1/chat/analyze |
| Method | POST |
| Content-Type | application/json |
| 说明 | 接收群聊消息数据，执行智能分析，返回结构化结果 |

### 3.2 数据来源说明

> **重要说明**：本接口接收的消息数据格式来源于企业微信群聊原始消息，由Java后端从企业微信API采集后推送至本服务。

**数据流向：**
```
企业微信群聊 → 企业微信API → Java后端采集 → Python分析服务
```

**各方职责：**

| 角色 | 职责 |
| :--- | :--- |
| 企业微信 | 产生群聊消息数据 |
| Java后端 | 从企业微信API采集数据，转换为标准格式，推送至分析服务 |
| Python分析服务 | 接收消息数据，执行智能分析，返回分析结果 |

**说明：**
- `messages` 数组中的字段格式由企业微信API定义
- Java后端负责数据的采集、清洗和格式转换
- Python分析服务按此格式接收数据并进行解析和分析
- 字段均为可选，系统会进行容错处理

### 3.3 请求参数

#### 请求体结构

```json
{
    "room_id": "wrU8vKSQAA_CQbRrGE303biSrw3uI-Rw",
    "room_name": "XX客户售后服务群",
    "analysis_type": ["sentiment", "sensitive", "summary", "highfreq"],
    "messages": [
        {
            "seq": 8389501,
            "msgid": "10950329173066564579_1776090443213_external",
            "action": "send",
            "from": "gaopengbo",
            "tolist": "[\"wmU8vKSQAAi9JgN3sIZMyU5fKFf_Zfeg\", \"15356147234\"]",
            "roomid": "wrU8vKSQAA_CQbRrGE303biSrw3uI-Rw",
            "msgtime": "2026-04-13 22:27:18",
            "msgtype": "text",
            "content": "{\"content\": \"其实如果咱们宿主占比不高的话 影响不是很大的\"}",
            "truename": "高鹏博",
            "avatar": "https://...",
            "position": "初级技术工程师",
            "job": "售后",
            "members": "[{\"group_nickname\":\"\",\"name\":\"高鹏博\",\"type\":1,\"userid\":\"gaopengbo\"}, {\"group_nickname\":\"\",\"name\":\"客户A\",\"type\":2,\"userid\":\"customer001\"}]"
        }
    ]
}
```

#### 参数说明

**根级参数：**

| 参数名 | 类型 | 必填 | 说明 |
| :--- | :--- | :--- | :--- |
| room_id | string | **是** | 群聊唯一标识 |
| room_name | string | 否 | 群名称，用于结果展示 |
| analysis_type | array | 否 | 分析类型列表，可选值见下表。不传或传空数组则执行全部分析 |
| messages | array | **是** | 消息数组，单次请求最多5000条 |

**analysis_type 可选值：**

| 值 | 说明 |
| :--- | :--- |
| sentiment | 情感分析 |
| sensitive | 敏感词检测 |
| summary | 摘要生成 |
| highfreq | 高频词统计 |
| unanswered | 漏回检测（检测客户消息是否被员工漏回） |

**messages 数组元素参数：**

| 参数名 | 类型 | 必填 | 说明 |
| :--- | :--- | :--- | :--- |
| seq | integer | 否 | 消息序号 |
| msgid | string | 否 | 消息唯一ID |
| action | string | 否 | 动作类型(send/recall等) |
| from | string | 否 | 发送者用户ID |
| tolist | string | 否 | 接收者列表(JSON字符串) |
| roomid | string | 否 | 群聊ID |
| msgtime | string | 否 | 消息时间，格式：yyyy-MM-dd HH:mm:ss |
| msgtype | string | 否 | 消息类型：text/voice/image/file等 |
| content | string | 否 | 消息内容，通常为JSON字符串 |
| resultAsrTime | string | 否 | 语音识别时间 |
| resultContent | string | 否 | 语音转文字结果 |
| filepath | string | 否 | 文件路径 |
| openId | string | 否 | 用户OpenID |
| truename | string | 否 | 发送者真实姓名 |
| avatar | string | 否 | 发送者头像URL |
| position | string | 否 | 发送者职位 |
| job | string | 否 | 发送者职责(销售/售后等) |
| members | string | 否 | 群成员列表(JSON字符串) |
| contentItem | string | 否 | 图片OCR内容等 |
| reOpenId | string | 否 | 回复消息的发送者ID |
| reTruename | string | 否 | 回复消息的发送者姓名 |
| reAvatar | string | 否 | 回复消息的发送者头像URL |
| rePosition | string | 否 | 回复消息的发送者职位 |
| isSingle | string | 否 | 是否单聊 |
| rawJson | string | 否 | 原始JSON数据 |

**members 字段格式说明：**

members是一个JSON字符串，解析后为数组，每个元素包含：

| 字段 | 类型 | 说明 |
| :--- | :--- | :--- |
| userid | string | 用户ID |
| name | string | 用户姓名 |
| group_nickname | string | 群昵称 |
| type | integer | 用户类型：1=员工，2=客户 |
| join_time | integer | 入群时间戳 |

**content 字段格式说明：**

文本消息的content通常为JSON字符串，格式如：
```json
{"content": "实际消息文本内容"}
```

## 4. 批量分析接口（推荐）

### 4.1 接口信息

| 项目 | 值 |
| :--- | :--- |
| URL | /api/v1/chat/batch-analyze |
| Method | POST |
| Content-Type | application/json |
| 说明 | 批量分析多个群聊消息，支持并发处理，适合大规模群聊分析场景 |

### 4.2 使用场景

**推荐使用批量接口的场景：**
- 需要同时分析多个群聊（如每天定时分析所有客户群）
- 群聊数量较多（>10个）
- 需要减少HTTP请求开销
- 希望利用并发处理提高效率

**性能优势：**
- 一次HTTP请求完成所有群的分析
- 内部并发处理，默认5个群同时分析
- 自动统计成功/失败数量
- 返回总耗时信息

### 4.3 请求参数

#### 请求体结构

```json
{
    "rooms": [
        {
            "room_id": "wrU8vKSQAA_CQbRrGE303biSrw3uI-Rw",
            "room_name": "XX客户售后服务群",
            "messages": [
                {
                    "seq": 8389501,
                    "msgid": "10950329173066564579",
                    "from": "gaopengbo",
                    "roomid": "wrU8vKSQAA_CQbRrGE303biSrw3uI-Rw",
                    "msgtime": "2026-04-13 22:27:18",
                    "msgtype": "text",
                    "content": "{\"content\": \"消息内容\"}",
                    "truename": "高鹏博",
                    "job": "售后"
                }
            ]
        },
        {
            "room_id": "wrU8vKSQAA_CQbRrGE303biSrw3uI-Rx",
            "room_name": "YY客户销售群",
            "messages": [...]
        }
    ],
    "analysis_type": ["sentiment", "sensitive", "summary", "highfreq"],
    "max_concurrent": 10
}
```

#### 参数说明

**根级参数：**

| 参数名 | 类型 | 必填 | 说明 |
| :--- | :--- | :--- | :--- |
| rooms | array | **是** | 多个群的数据列表，建议单次不超过100个群 |
| analysis_type | array | 否 | 分析类型列表，不传则执行全部分析 |
| max_concurrent | integer | 否 | 最大并发处理数，默认10，建议范围5-20 |

**rooms 数组元素参数：**

| 参数名 | 类型 | 必填 | 说明 |
| :--- | :--- | :--- | :--- |
| room_id | string | **是** | 群聊唯一标识 |
| room_name | string | 否 | 群名称 |
| messages | array | **是** | 该群的消息数组，格式同单个分析接口 |

### 4.4 响应参数

#### 成功响应示例

```json
{
    "code": 0,
    "message": "success",
    "data": {
```

## 7. 常见问题

### 7.1 消息数量限制

**Q: 单次请求最多支持多少条消息？**

A: 单次请求最多支持5000条消息。如果消息数量超过限制，建议按时间范围分批发送。

### 7.2 请求体大小限制

**Q: 请求体大小有限制吗？**

A: 请求体大小限制为10MB。超过限制时服务会返回错误码40001，需要分批发送。

### 7.3 分析类型选择

**Q: 如何只执行部分分析功能？**

A: 通过analysis_type参数指定需要执行的分析类型：
```json
{
    "room_id": "xxx",
    "analysis_type": ["sentiment", "sensitive"],
    "messages": [...]
}
```
不传或传空数组则执行全部分析。

### 7.4 响应时间

**Q: 接口响应时间大概是多少？**

A: 响应时间取决于消息数量和启用的分析类型：
- 100条消息，全部分析：约3-5秒
- 500条消息，全部分析：约10-15秒
- 仅敏感词检测：约1-2秒

### 7.5 LLM降级

**Q: LLM服务不可用时接口会失败吗？**

A: 不会。系统有降级策略：
- 情感分析：LLM不可用时使用规则匹配结果
- 摘要生成：LLM不可用时返回null
- 漏回检测：LLM不可用时默认判定为低风险

### 7.6 并发限制

**Q: 服务支持多大并发？**

A: 建议并发控制在每分钟10个请求以内。如需更高并发，请联系运维扩容。

### 7.7 字段为空的处理

**Q: 请求中某些字段为空会影响分析吗？**

A: 系统对空字段有容错处理：
- members为空：无法识别客户/员工角色，标记为"未知"
- content为空：跳过该消息
- job为空：根据members中的type推断角色

### 7.8 漏回检测功能

**Q: 什么是漏回检测？**

A: 漏回检测(unanswered)分析群聊尾部记录，判断客户发送的消息是否被员工漏回。如果客户提出了明确的问题、反馈了报错或不满、催促进度等，而员工未回复，则判定为漏回并给出风险等级和建议。

### 7.9 高频词业务需求提取

**Q: 高频词分析现在是什么逻辑？**

A: 高频词分析已从简单的词频统计升级为业务需求提取。通过LLM识别客户提及的生信分析业务需求，并与标准业务库进行匹配，返回标准化的业务名称和客户的原始表述。例如客户说"跑个差异"、"做差异"会被映射为"差异表达与丰度分析"。

### 7.10 调试建议

**Q: 如何调试接口问题？**

A: 建议：
1. 使用Swagger UI（/docs）进行接口测试
2. 检查请求体JSON格式是否正确
3. 确认Content-Type为application/json
4. 查看响应体中的message字段获取错误详情

## 8. 看板下钻与明细导出

### 8.1 分页预览明细

```http
GET /api/v1/dashboard/drilldown
```

通用查询参数：

| 参数 | 必填 | 说明 |
| :--- | :--- | :--- |
| target | 是 | 服务端白名单中的下钻目标，例如 `summary.total_groups`、`business.region` |
| measure | 是 | 统计单位，例如 `project_count`、`group_count`、`message_count` |
| period | 否 | `today/week/month/quarter/year/custom`，默认 `month` |
| start_date、end_date | 自定义周期时是 | `YYYY-MM-DD` |
| region、aftersaler、category、key_account | 否 | 与看板顶部筛选一致 |
| dimension_value | 否 | 点击的数据点，如日期、区域、售后人员或产品分类 |
| secondary_value | 否 | 交叉分析的第二维度值 |
| search | 否 | 在当前下钻结果中搜索，最大 100 字符 |
| page、page_size | 否 | 默认第 1 页、每页 20 条；每页最大 100 条 |

响应 `data` 包含：

- `title`、`level`、`definition`：明细标题、项目/群聊/消息等层级及统计口径。
- `period`、`filters`：本次下钻固定使用的日期和业务筛选快照。
- `reconciliation`：`dashboard_value`、`detail_value`、`comparable`、`consistent` 和核对说明。
- `columns`、`items`：动态列定义和当前页数据。
- `total`、`page`、`page_size`、`total_pages`：分页信息。

非法的 `target` 或不属于目标白名单的 `measure` 返回 HTTP 422，不会被拼接为数据库字段或 SQL。

### 8.2 导出当前明细

```http
GET /api/v1/dashboard/drilldown/export
```

参数与预览接口一致，但不接收分页参数。响应为 XLSX 文件，包含：

- `导出说明`：统计口径、日期、业务筛选、点击维度、看板值、明细数、核对结果和生成时间。
- `明细数据`：与预览相同字段和脱敏规则下的全部匹配记录。

单次最大导出行数由环境变量 `DASHBOARD_DRILLDOWN_EXPORT_MAX_ROWS` 控制，默认 100000。超过上限返回 HTTP 422，调用方应缩小日期或业务筛选范围；服务端不会静默截断。

### 8.3 当前支持的目标

- 经营摘要：服务群、消息、项目群、区域、售后、产品分类、重点客户。
- 沟通洞察：每日趋势、时段分布、Top 群、活跃周期、高频主题。
- 业务覆盖：区域、售后、产品、重点客户、项目年份、异常交付项目。
- 服务质量：待回复、已响应/无待回复、客户正负向、服务正负向。
- 数据质量：未提取项目号、LIMS 未返回、无 LIMS 关联、映射回退/冲突、重复分析记录。
