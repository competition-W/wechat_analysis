## 1. 项目概述

### 1.1 项目背景

企业微信客户群聊智能分析系统是一个基于大语言模型（LLM）的智能分析服务，用于对企业微信群聊内容进行自动化、结构化分析。系统接收Java后端推送的群聊消息数据，经过智能分析后返回结构化结果。

### 1.2 核心功能

| 功能模块 | 描述 |
| :--- | :--- |
| 情感分析 | 对消息进行**角色感知**的情感倾向分析，区分客户好评/差评与员工积极/恶劣态度 |
| 敏感词检测 | 检测消息中是否包含预设敏感词，返回命中详情 |
| 摘要生成 | 根据群聊记录生成管理层视角的业务摘要，聚焦客户诉求、处理进展与待办事项 |
| 高频词统计 | 基于LLM提取客户生信分析业务需求意图，映射到标准业务名称并统计频次 |
| 漏回检测 | 分析群聊尾部消息，检测员工是否遗漏回复客户的重要诉求，支持正则快速拦截与LLM深度判定双引擎 |

## 2. 系统架构

### 2.1 整体架构

系统采用分层架构设计，各层职责明确：

| 层级 | 组件 | 职责 |
| :--- | :--- | :--- |
| API层 | FastAPI Routes | 请求校验、路由分发、异常处理、响应封装 |
| 服务层 | Services | 业务逻辑处理、模块编排、并发控制 |
| 模型层 | Models | 数据模型定义、序列化/反序列化 |
| 工具层 | Utils | LLM客户端封装、通用工具函数 |
| 配置层 | Config | 环境变量加载、配置管理 |

### 2.2 请求处理流程

```
请求 → 路由层 → 预处理模块 → 分析模块(并行) → 结果聚合 → 响应
                          ↓
              ┌──────────┼──────────┬──────────┐
              ↓          ↓          ↓          ↓
          情感分析   敏感词检测   摘要生成   高频词/漏回
              │          │          │          │
              └──────────┼──────────┼──────────┘
                         ↓
                    LLM调用(按需)
```

### 5.4 服务层 (services/)

#### 5.4.1 preprocessor.py - 预处理模块

**职责：**
- 解析原始消息JSON，提取纯文本
- 解析members字段构建群成员映射
- 补全发送者角色（销售/售后/客户）
- 输出标准化消息对象

**核心数据结构：**

```python
@dataclass
class MemberInfo:
    userid: str
    name: str = ""
    group_nickname: str = ""
    type: int = 1          # 1=员工, 2=客户
    join_time: int = 0

@dataclass
class NormalizedMessage:
    msgid: str
    seq: int
    roomid: str
    from_userid: str
    sender_name: str
    sender_position: str
    sender_job: str
    sender_role: str       # 客户/员工/未知
    text_content: str
    msgtime: str
    msgtype: str
    raw_content: str
    members: Dict[str, MemberInfo]
```

**角色推断逻辑：**

1. 从members字段获取用户type（1=员工，2=客户）
2. 根据job字段判断具体角色（销售/售后）
3. 无法确定时标记为"未知"

#### 5.4.2 sentiment.py - 情感分析模块（V1.1 角色感知版本）

**分析策略：**

```
标准化消息列表
    │
    ▼
┌──────────────────────────────────────┐
│         过滤非文本消息                 │
└──────────────┬───────────────────────┘
               │
               ▼
┌──────────────────────────────────────┐
│         规则扫描（角色感知）           │
│  ┌─────────────────┬────────────────┐│
│  │   客户角色       │   员工角色      ││
│  │ 命中差评词库     │ 命中恶劣态度词库 ││
│  │ → bad_review    │ → bad_attitude ││
│  │ 命中好评词库     │ 命中积极词库    ││
│  │ → good_review   │ → positive     ││
│  └─────────────────┴────────────────┘│
└──────────────┬───────────────────────┘
               │ 未命中（neutral）
               ▼
┌──────────────────────────────────────┐
│      LLM 并行批量分析（角色感知）       │
│  ┌─────────────────────────────────┐ │
│  │ ThreadPoolExecutor 并发处理      │ │
│  │ - 每批50条消息                   │ │
│  │ - 最大并发批次数: 10             │ │
│  └─────────────────────────────────┘ │
│  Prompt 包含角色标签，输出分类：        │
│  - 客户: good_review / bad_review     │
│  - 员工: positive / bad_attitude      │
│  - 中性: neutral                      │
└──────────────┬───────────────────────┘
               │
               ▼
┌──────────────────────────────────────┐
│      结果合并与按角色分发统计          │
│  ┌─────────────────┬────────────────┐│
│  │   客户统计       │   员工统计      ││
│  │ good_reviews    │ positive      ││
│  │ bad_reviews     │ bad_attitude  ││
│  └─────────────────┴────────────────┘│
└──────────────────────────────────────┘
```

**角色感知分类规则：**

| 角色 | 分类 | 说明 |
| :--- | :--- | :--- |
| 客户 | good_review | 好评：赞扬、感谢、满意 |
| 客户 | bad_review | 差评：抱怨、愤怒、催促、不满 |
| 员工/售后/销售 | positive | 积极：耐心解答、热情、主动推进 |
| 员工/售后/销售 | bad_attitude | 恶劣态度：推诿、不耐烦、敷衍、指责客户 |
| 所有 | neutral | 中性：纯客观陈述，无情感波动 |

**批处理优化：**

- 批量大小：50条消息/批
- 最大并发批次数：10
- 使用ThreadPoolExecutor并行处理

**关键参数：**

| 参数 | 值 | 说明 |
| :--- | :--- | :--- | :--- |
| BATCH_SIZE | 50 | 每批处理消息数 |
| MAX_CONCURRENT_BATCHES | 10 | 最大并发批次数 |

#### 5.4.3 sensitive.py - 敏感词检测模块

**检测策略：**

1. 使用AC自动机（pyahocorasick）构建敏感词匹配器
2. 对每条文本消息进行多模式匹配
3. 返回命中词列表及详情

**性能优化：**

- 应用启动时构建AC自动机，全请求复用
- 若pyahocorasick未安装，降级为字符串包含匹配

**数据结构：**

**摘要结构设计（V1.8 JSON数组版本）：**

| JSON字段 | 类型 | 说明 |
| :--- | :--- | :--- |
| overview | string | 用1-2句话高度概括今日群聊的核心议题或突发状况 |
| demands | string[] | 客户提出的核心问题、需求或咨询内容（字符串数组） |
| actions | string[] | 我方员工的响应动作、提供的解决方案或达成的共识（字符串数组） |
| todos | string[] | 待办事项及责任人，只提取明确的时间节点（字符串数组） |

**采样优先级：**

| 优先级 | 条件 | 说明 |
| :--- | :--- | :--- |
| 高 | 包含负面关键词 | 问题、错误、失败、投诉等 |
| 中 | 包含疑问词 | ?、？、怎么、为什么等 |
| 低 | 其他消息 | 均匀采样 |

**重试机制：**

使用tenacity实现指数退避重试：
- 最大重试次数：2次
- 等待时间：指数增长，最小1秒，最大10秒

#### 5.4.6 unanswered.py - 漏回检测模块（V1.4 双引擎版本）

**核心功能：**

分析群聊尾部消息，检测员工是否遗漏回复客户的重要诉求。采用"正则快速拦截 + LLM深度判定"双引擎架构，兼顾效率与准确率。

**处理流程：**

```
标准化消息列表
    │
    ▼
┌──────────────────────────────────────┐
│         尾部截断算法                   │
│  1. 倒序遍历找到最后一条员工消息        │
│  2. 截取其后所有客户消息作为待定池      │
│  3. 若无员工消息，取所有客户消息        │
└──────────────┬───────────────────────┘
               │
               ▼
        ┌──────────────┐
        │ 待定池为空？  │
        └──────┬───────┘
               │
       ┌───────┴───────┐
       │ 是            │ 否
       ▼               ▼
   ┌────────┐  ┌──────────────────────┐
   │无漏回  │  │ 引擎A：正则快速拦截   │
   │(默认)  │  │ 匹配结束客套话        │
   └────────┘  └──────────┬───────────┘
                           │
                   ┌───────┴───────┐
                   │ 全部匹配？     │
                   └───────┬───────┘
                           │
                   ┌───────┴───────┐
                   │ 是            │ 否
                   ▼               ▼
               ┌────────┐  ┌──────────────────┐
               │无漏回  │  │ 引擎B：LLM深度判定│
               │(闭环)  │  │ 分析客户意图      │
               └────────┘  │ tenacity重试机制  │
                           └────────┬─────────┘
                                    │
                    ┌───────────────┼───────────────┐
                    │               │               │
                    ▼               ▼               ▼
              ┌──────────┐  ┌──────────┐  ┌──────────┐
              │ is_missed│  │ is_missed│  │ 降级结果  │
              │ = false  │  │ = true   │  │ 人工核查  │
              └──────────┘  │ missed_  │  └──────────┘
                            │ messages │
                            └──────────┘
```

**尾部截断算法：**

1. 倒序遍历当天所有标准化消息
2. 找到最后一条由我方员工（`sender_role` 在 `['售后', '员工', '销售']` 中）发出的消息
3. 截取该员工消息**之后**的所有消息，构成"待定漏回消息池"
4. 如果待定池为空，说明最终由员工兜底结语，直接判定为无漏回

**引擎 A：正则快速拦截（Fast Path）：**

针对待定池中的客户消息进行正则匹配。如果**所有**尾部客户消息都仅仅是结束客套话，则判定为正常闭环，无漏回。

参考正则：
```python
r'^(好的|好滴|好的呢|收到|谢谢|感谢|辛苦|辛苦了|麻烦了|知道了|ok|OK|1|👍|\[抱拳\]|\[玫瑰\])[~～！!。]*$'
```

**引擎 B：LLM 深度判定（Deep Path）：**

如果正则无法完全拦截，说明尾部存在长文本或潜在业务问题，将这些尾部消息按时间正序打包，发送给 LLM 进行意图判定。

**内置 LLM Prompt：**

```
你是一个企业客户服务质量监督员。请分析以下截取的【群聊对话尾部记录】（全部由客户发送），判断我方员工是否漏回了客户的重要消息，并严格输出 JSON 格式。

【判定标准】：
1. 正常结束（无漏回）：如果客户发送的仅仅是确认、感谢、客套话（如"好的"、"谢谢"、"麻烦了"、"知道了"）或是毫无意义的表情/符号，表明当前沟通已自然闭环，不需要我方继续回复。
2. 存在漏回：如果客户提出了明确的问题（如"怎么弄？"）、提出了新需求、反馈了报错/不满、或者在催促进度，则表明我方必须回复但未回复。

【群聊尾部记录】：
{trailing_messages}

【输出格式要求】：
请直接输出 JSON 格式（不要有任何 Markdown 标记或多余文本），字段如下：
{
  "is_missed": true 或 false,
  "risk_level": "high" 或 "low",
  "suggested_action": "一句话建议，比如'需排查XX报错问题'。如果没有漏回，填空字符串"

```

**输出数据结构：**

```python

    "is_missed": False,           # 是否存在漏回
    "risk_level": "low",          # 风险等级: high/low
    "missed_messages": [          # 漏回消息详情列表
        {
            "msgid": "xxx",
            "sender_name": "客户张三",
            "msgtime": "2026-04-21 18:30:00",
            "content": "这个问题怎么还没解决？"
        }
    ],
    "suggested_action": "需跟进客户报错问题"  # 建议行动
}
```

**异常降级策略：**

| 场景 | 降级策略 |
| :--- | :--- |
| LLM调用失败 | 返回 `is_missed=True`, `risk_level="low"`, `suggested_action="系统判定降级，请人工核查是否漏回。"` |
| JSON解析失败 | 同上，并带上截取到的消息详情 |
| 无消息数据 | 返回默认值 `is_missed=False` |

使用tenacity实现指数退避重试：
- 最大重试次数：2次
- 等待时间：指数增长，最小1秒，最大10秒

1. 意图提取：识别群聊中客户真正需要的分析动作。必须强制忽略纯粹的闲聊、无意义数字、语气词。
2. 标准化映射（核心）：

【输出格式】：

## 12. 性能优化

### 12.1 问题背景

在批量群聊分析场景下，系统遭遇严重性能瓶颈。测试显示，分析 62 个群聊（约 1000 条消息）耗时超过 20 分钟。

**核心问题诊断：**

| 问题 | 描述 | 影响 |
| :--- | :--- | :--- |
| Event Loop 阻塞 | FastAPI 是单线程异步框架，但同步网络请求（LLM API 调用）阻塞了整个事件循环 | 所有请求排队等待，吞吐量骤降 |
| 缺乏内部并发 | 批量接口中，各分析模块（情感、敏感词、摘要、高频词、漏回）串行执行 | 单群分析时间累加 |
| 无效微小请求 | 大量 1-2 条消息的群聊被全量送入 LLM 分析 | 浪费时间和网络开销 |

### 12.2 优化方案

#### 12.2.1 asyncio.to_thread 释放事件循环

**原理：** FastAPI 基于单线程事件循环，同步阻塞调用会卡死整个服务。`asyncio.to_thread` 将同步函数投入线程池执行，释放主事件循环处理其他请求。

**改造前（阻塞事件循环）：**

```python
# 同步调用，阻塞整个事件循环
sentiment_result = sentiment_analyzer.analyze(normalized)
```

**改造后（真正的异步）：**

```python
# 投入线程池执行，释放事件循环
sentiment_result = await asyncio.to_thread(sentiment_analyzer.analyze, normalized)
```

**覆盖范围：** 所有底层分析服务调用均使用 `asyncio.to_thread` 包裹：

| 服务 | 方法 | 改造状态 |
| :--- | :--- | :--- |
| Preprocessor | process() | ✅ 已改造 |
| SentimentAnalyzer | analyze() | ✅ 已改造 |
| SensitiveWordDetector | detect() | ✅ 已改造 |
| SummaryGenerator | generate() | ✅ 已改造 |
| HighFreqAnalyzer | analyze() | ✅ 已改造 |
| UnansweredAnalyzer | analyze() | ✅ 已改造 |

#### 12.2.2 模块级并发

**单群组分析并发架构：**

```
单个群聊分析请求
    │
    ▼
┌─────────────────────────────────────────────────────────┐
│                    asyncio.gather                        │
│  ┌─────────┬─────────┬─────────┬─────────┬─────────┐   │
│  │情感分析  │敏感词   │摘要生成  │高频词   │漏回检测  │   │
│  │to_thread│to_thread│to_thread│to_thread│to_thread│   │
│  └────┬────┴────┬────┴────┬────┴────┬────┴────┬────┘   │
│       │         │         │         │         │        │
│       └─────────┴─────────┴─────────┴─────────┘        │
│                         │                                │
│                         ▼                                │
│              线程池并行执行（5个任务）                     │
└─────────────────────────────────────────────────────────┘
    │
    ▼
结果聚合返回
```

**关键代码：**

```python
sentiment_result, sensitive_result, summary_result, highfreq_result, unanswered_result = await asyncio.gather(
    run_sentiment(),    # 内部使用 asyncio.to_thread
    run_sensitive(),
    run_summary(),
    run_highfreq(),
    run_unanswered(),
)
```

#### 12.2.3 短路拦截机制

针对"碎片请求"（消息量极少的群聊），在调用 LLM 前进行短路判断，直接返回默认结果。

**摘要短路：**

```python
SUMMARY_SHORT_CIRCUIT_THRESHOLD = 5  # 消息数阈值

async def run_summary():
    if AnalysisType.SUMMARY in analysis_types:
        if msg_count < SUMMARY_SHORT_CIRCUIT_THRESHOLD:
            # 短路返回默认摘要，跳过 LLM 调用
            return "群内互动较少，暂无核心议题。"
        return await asyncio.to_thread(summary_generator.generate, normalized)
    return None
```

**高频词短路：**

```python
HIGHFREQ_SHORT_CIRCUIT_MSG_THRESHOLD = 3      # 消息数阈值
HIGHFREQ_SHORT_CIRCUIT_CHARS_THRESHOLD = 50   # 总字数阈值

async def run_highfreq():
    if AnalysisType.HIGHFREQ in analysis_types:
        if msg_count < HIGHFREQ_SHORT_CIRCUIT_MSG_THRESHOLD:
            return HighFreqResult(words=[])  # 短路返回空列表
        if total_chars < HIGHFREQ_SHORT_CIRCUIT_CHARS_THRESHOLD:
            return HighFreqResult(words=[])
        return await asyncio.to_thread(highfreq_analyzer.analyze, normalized)
    return None
```

**短路阈值配置：**

| 模块 | 阈值条件 | 默认返回 |
| :--- | :--- | :--- |
| 摘要 | 消息数 < 5 | `"群内互动较少，暂无核心议题。"` |
| 高频词 | 消息数 < 3 或 总字数 < 50 | `HighFreqResult(words=[])` |

#### 12.2.4 Semaphore 并发控制

保留群组级并发控制，防止瞬间并发过高触发 LLM API 限流（HTTP 429）。

```python
semaphore = asyncio.Semaphore(request.max_concurrent or 5)

async def analyze_single_room(room: RoomData):
    async with semaphore:  # 限制同时处理的群聊数量
        # 群内模块级并发
        ...
```

**并发层级示意：**

```
批量分析请求（62个群聊）
    │
    ▼
┌─────────────────────────────────────────────────────────┐
│           Semaphore(5) - 最多5个群聊并发处理              │
│  ┌─────────┐ ┌─────────┐ ┌─────────┐ ┌─────────┐ ┌───┐ │
│  │ 群聊 1  │ │ 群聊 2  │ │ 群聊 3  │ │ 群聊 4  │ │...│ │
│  │ 5模块并发│ │ 5模块并发│ │ 5模块并发│ │ 5模块并发│ │   │ │
│  └─────────┘ └─────────┘ └─────────┘ └─────────┘ └───┘ │
└─────────────────────────────────────────────────────────┘
    │
    ▼
其余群聊排队等待（FIFO）
```

### 12.3 容错设计

每个子任务独立捕获异常，单个模块失败不影响整个群聊分析：

```python
async def run_sentiment():
    if AnalysisType.SENTIMENT in analysis_types:
        try:
            return await asyncio.to_thread(sentiment_analyzer.analyze, normalized)
        except Exception as e:
            logger.error(f"情感分析失败: {e}")
            return None  # 返回 None，不中断其他模块
    return None
```

### 12.4 性能对比

| 指标 | 优化前 | 优化后 | 提升 |
| :--- | :--- | :--- | :--- |
| 62群聊分析耗时 | ~20 分钟 | ~1-2 分钟 | **10-20x** |
| 单群平均耗时 | ~20 秒 | ~1-2 秒 | **10-20x** |
| 事件循环阻塞 | 严重 | 无 | 完全解决 |
| 模块执行方式 | 串行 | 并行 | 5x 理论加速 |
| 碎片请求处理 | 全量 LLM | 短路跳过 | 大幅减少无效调用 |

### 12.5 最佳实践

1. **识别阻塞点：** 使用 `asyncio.to_thread` 包裹所有同步 I/O 操作
2. **分层并发：** 群组级 Semaphore + 模块级 gather，两层并发
3. **短路优化：** 对微小请求提前返回，避免无效 LLM 调用
4. **容错隔离：** 每个子任务独立 try-except，失败不扩散
5. **并发控制：** Semaphore 防止 API 限流，平衡吞吐与稳定性
